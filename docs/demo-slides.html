Background

From Static Calculations To A Runtime System

  • The project turns a calculation into a living network of topics and agents.
  • Values move through the graph as messages, not as one hardcoded formula.
  • Changing the config changes runtime behavior without recompiling.
  • The browser becomes a control room: deploy, publish, edit, inspect, reset.
ProblemHardcoded computation is difficult to inspect and change.
IdeaRepresent computation as a configurable message graph.
ResultA live console that shows values flowing through the graph.

Design

Design: Small Components, Clear Ownership

Browser UIUpload, edit, publish, inspect
→
HTTP ServerSocket accept loop, request pool, longest-prefix routing
→
ServletsSmall route handlers for config, graph, topics, and static HTML
GenericConfigReads three-line blocks and loads agents by reflection
→
Topic GraphTopics keep latest messages, publishers, and subscribers
→
ParallelAgentSerializes callbacks on a worker thread and owns shutdown
SOLIDServer, parser, servlets, config, graph, and view each have one job.
Reusable APIThe HTTP server API is documented with Javadocs for other developers.
LifecycleConfig replacement, reset, and close paths release registered resources.

Main Flow To Record

Browser Console Features

  • Add prebuilt math and Boolean agents.
  • Add and remove legal graph edges.
  • Reset active graph state with one action.
  • Publish only to source topics to protect computed values.

Beyond The Minimum

Extra Engineering Work

  • Non-math processing with Boolean gates and majority logic.
  • Gate-shaped SVG nodes instead of generic agent bubbles.
  • Variable-input and multi-output agents for larger graphs.
  • Legal edge editing with duplicate publisher protection.
  • Cycle-aware layout for feedback graphs.
  • Bounded `ParallelAgent` shutdown under full queues.

Summary

What I Learned And Will Take Forward

  • How HTTP works under a framework: sockets, parsing, routing, responses.
  • How to model a runtime system with small interfaces and clear ownership.
  • How reflection can make a plugin-like configuration flow.
  • Why concurrency lifecycle and shutdown paths matter.
  • How backend state can become a useful visual tool.
